Manual Repository Setup Guide

Complete step-by-step instructions for creating RevNova-Dev and RevNova-Test repositories using GitHub's web interface. Perfect for one-time setup with temporary account access.

Overview

This guide is designed for a developer who will have one-time temporary access to the owner's GitHub account to set up the development and testing repositories. No command-line tools or technical installation required — everything is done through GitHub's web interface.

Time Required: Approximately 30-45 minutes
Prerequisites: Web browser + temporary access to naimishkathrani-web GitHub account
Outcome: Two new repositories (RevNova-Dev, RevNova-Test) with proper settings and protection

RevNova-Dev Development environment (daily work) https://github.com/naimishkathrani-web/RevNova-Dev

RevNova Production (already exists - no changes needed) https://github.com/naimishkathrani-web/RevNova

  4. Select "New repository" from the dropdown menu

Step 1.2: Configure Repository Settings

Fill out the new repository form with these exact values:

Field Value

Repository name RevNova-Dev (case-sensitive, exact spelling)
Description RevNova Development Repository - Daily development work
Visibility Public (select the "Public" radio button)

3 Configure Branch Protection for RevNova-Dev

Branch protection prevents accidental direct commits to the main branch and enforces code review via pull requests.

Step 3.1: Navigate to Branch Settings

  1. Go to: https://github.com/naimishkathrani-web/RevNova-Dev
  2. Click "Settings" tab (near the top of the page, right side)
  3. In the left sidebar, click "Branches" under "Code and automation"
  4. Under "Branch protection rules", click "Add branch protection rule"
Screenshot placeholder: Settings → Branches → Add branch protection rule button

Step 3.2: Configure Protection Rule

Configure the following settings:

Branch Name Pattern

  • Branch name pattern: main

Protect Matching Branches

Check/enable these options:

  • ✓ Require a pull request before merging
    • ✓ Require approvals: 1 (change number to 1)
    • ✓ Dismiss stale pull request approvals when new commits are pushed
  • ✓ Require status checks to pass before merging
    • ✓ Require branches to be up to date before merging
    • (Leave status checks empty for now - will be added when CI/CD is set up)
  • ✓ Require conversation resolution before merging
Do NOT enable: "Do not allow bypassing the above settings" (leave unchecked) - this allows admins to push if needed during setup.

Step 3.3: Save Protection Rule

  1. Scroll to the bottom of the page
  2. Click the green "Create" button
✓ Checkpoint: You should see "main" listed under Branch protection rules with a green checkmark.

5 Add Developer Collaborators to RevNova-Dev

Only the Development repository (RevNova-Dev) needs collaborators. The owner will control access to Test and Production repositories.

Step 5.1: Navigate to Collaborators

  1. Go to: https://github.com/naimishkathrani-web/RevNova-Dev
  2. Click "Settings" tab
  3. In the left sidebar, click "Collaborators and teams" (under "Access")
  4. You may need to confirm your password/2FA

Step 5.2: Add Each Developer

You will add 3 developers. For each developer:

  1. Click the green "Add people" button
  2. In the search box, type the developer's GitHub username or email
  3. Select the correct user from the dropdown
  4. Click "Add [username] to this repository"
  5. In the permission dropdown, select "Write" (not Admin, not Read)
  6. The developer will receive an invitation email (they must accept it)
Developer Usernames: The owner should provide you with the exact GitHub usernames for the 3 developers. If usernames are unknown, ask the owner before proceeding.

Repeat this process 3 times - once for each developer:

  • Developer 1 (Backend/Database): ___________ (owner to fill in)
  • Developer 2 (Frontend/React): ___________ (owner to fill in)
  • Developer 3 (DevOps/Queue/QA): ___________ (owner to fill in)

Step 5.3: Verify Collaborators

  1. You should now see all 3 developers listed under "Manage access"
  2. Each should have "Write" permission (not "Admin" or "Read")
  3. Status will show "Pending" until they accept the invitation
✓ Checkpoint: 3 collaborators added to RevNova-Dev with Write access.

7 Push Initial Code to New Repositories

Note: This step requires command-line access. If you don't have Git installed or don't have access to the local RevNovaRepository folder, skip this section and inform the owner to complete it.

Step 7.1: Add Git Remotes (Command Line)

If you have access to the local repository folder on the owner's machine:

cd C:\Dev\RevNovaRepository

# Add dev remote
git remote add dev https://github.com/naimishkathrani-web/RevNova-Dev.git

# Add test remote
git remote add test https://github.com/naimishkathrani-web/RevNova-Test.git

# Verify remotes
git remote -v

Step 7.2: Push Code to Dev and Test

# Push to dev
git push -u dev main

# Push to test
git push -u test main
✓ Result: Both RevNova-Dev and RevNova-Test now contain the same code as Production.

Alternative: Owner Can Push Later

If you cannot access the command line, inform the owner to run these commands after you complete the setup:

git remote add dev https://github.com/naimishkathrani-web/RevNova-Dev.git
git remote add test https://github.com/naimishkathrani-web/RevNova-Test.git

git push -u test main

Troubleshooting

Problem: Repository name is taken

Solution: If you see an error "Repository name already exists", contact the owner immediately. They may already have a repository with that name.

Problem: Can't find "Add branch protection rule" button

Solution: Make sure you are in Settings → Branches. The button is below the "Default branch" section.

Problem: Can't add collaborators

Solution: Ensure you are logged in as the owner (naimishkathrani-web). Only the owner can add collaborators.

Problem: Developer usernames unknown

Solution: Ask the owner for the exact GitHub usernames before proceeding with Step 5.

Problem: Git push fails with "repository not found"

Solution: Verify the repository URLs are correct and that you created the repositories in Step 1 and 2. The repositories must exist before you can push to them.
