RevNova Onboarding

Everything you need to set up your machine, access the correct repositories, configure VS Code + GitHub Copilot, connect to PostgreSQL, and start building confidently.

Project Overview

RevNova is an AI-powered platform for enterprise system transformation: accelerating SFDC CPQ → Revenue Cloud migrations and enabling AI-driven product configuration and planning. Insights consolidated from product briefs (Gemini/DeepSeek inputs) are captured across our requirements.

  • Core Value: Reduce migration timelines by ~80% with high accuracy and repeatability.
  • Architecture: Object-oriented, source-agnostic 7-step wizard + reusable hybrid staging (Vanilla + EAV).
  • Frontend: React 18 + TypeScript + TailwindCSS (Wizard UI, dashboards).
  • Backend: Node.js 20 + Express + TypeScript (REST APIs).
  • Database: PostgreSQL 15/16 (multi-tenant), STG1 → STG2 → Target validation pipeline.
  • Queue: Bull (Redis 7) for ETL and validation jobs with backoff + idempotency.
  • AI: OpenAI GPT-4 for mapping suggestions and plan generators; embeddings + cosine similarity.
  • Storage: AWS S3/Azure Blob for artifacts, logs, and exports.

Requirements are per-phase and per-view. Start at Requirements Home.

Environment Strategy & Repositories

We use three repositories to separate concerns by environment. You will not clone the production repository. You will work in the Development repository.

  1. Production: naimishkathrani-web/RevNova (protected) — release-only, tagged versions.
  2. Testing: naimishkathrani-web/RevNova-Test (staging QA) — integration testing, pre-release branches.
  3. Development: naimishkathrani-web/RevNova-Dev (your working repo) — daily development.
Action for Owner: Create RevNova-Dev and RevNova-Test repos under the same owner, enable branch protections, and grant developer access to RevNova-Dev. Configure GitHub Environments (Dev/Test/Prod) and required status checks.
  • Flow: feature branches in Dev → PR to Dev main → sync to Test via PR → release to Prod via PR and tags.
  • Remotes setup: on your local, add origin = RevNova-Dev and test = RevNova-Test for promotion.

1) Machine Prerequisites (Windows) — Download & Install

  • VS Code: https://code.visualstudio.com/ — install for Windows (User Installer). After install, sign in to GitHub inside VS Code.
  • Node.js 20.x: https://nodejs.org/en/download — choose Windows 64-bit LTS 20.x; verify with node -v.
  • Git: https://git-scm.com/download/win — enable “Git from the command line”; set default editor to VS Code.
  • Python 3.11+ (for tooling): https://www.python.org/downloads/windows/ — add to PATH during install.
  • Docker Desktop: https://www.docker.com/products/docker-desktop/ — optional for local PostgreSQL/Redis.
  • PostgreSQL: psql included with server or install pgAdmin 4: https://www.pgadmin.org/download/; or Azure Data Studio + PostgreSQL extension: https://aka.ms/azuredatastudio.
  • Redis (optional): use Docker image redis:7-alpine.
  • GitHub CLI (gh): https://github.com/cli/cli — simplifies auth and PRs.

2) Access & Clone (Development Repo only)

  • Get access to RevNova-Dev. Do not clone the Production repo.
  • Authenticate: gh auth login in a terminal, or sign in via VS Code.
  • SSH Key (optional): generate with ssh-keygen and add to GitHub → Settings → SSH and GPG keys.
  • Clone: choose a workspace path like C:\\Dev and clone RevNova-Dev into RevNovaRepository.
  • Remotes: add test remote pointing to RevNova-Test for promotions.
  • Branching: feature/<area>-<desc> off main. Use small, focused commits.

3) VS Code Setup + GitHub Copilot

  • Open RevNovaRepository in VS Code. Set terminal to Windows PowerShell.
  • Install extensions: GitHub Copilot, GitHub Copilot Chat, ESLint, Prettier, GitLens, Tailwind CSS IntelliSense, Docker, YAML, REST Client/Thunder Client, Python, Pylance.
  • Enable Copilot: sign in to GitHub in VS Code → Extensions → GitHub Copilot → enable for workspace. Use Ctrl+Enter for inline suggestions.
  • Configure format-on-save and ESLint autofix in VS Code Settings.

Shared PostgreSQL Connection

We maintain a team PostgreSQL instance for development. Use read-only/restricted credentials unless you are assigned DDL tasks.

  1. Open your DB client (pgAdmin/Azure Data Studio/VS Code SQL extension).
  2. Create a new connection using the parameters below.
Connection Template (replace placeholders):
  • Host: <PG_HOST>
  • Port: 5432
  • Database: revnova_dev
  • SSL: require if mandated by host
  • User: <PG_USER_RO> (read-only)
  • Password: <PG_PASSWORD_RO>

Tip: If your VS Code has an existing saved connection, copy the parameters from the Database pane or your secrets manager. Do not commit secrets.

Local option: run PostgreSQL via Docker for offline work; keep a separate local database name like revnova_local.

Local Services (Optional)

  • Run PostgreSQL/Redis locally using Docker for offline work.
# Example Docker (optional)
# docker run -p 5432:5432 -e POSTGRES_PASSWORD=postgres -e POSTGRES_DB=revnova_dev postgres:16
# docker run -p 6379:6379 redis:7-alpine

Use environment variables for local secrets: PGHOST, PGUSER, PGPASSWORD, PGDATABASE, PGPORT, REDIS_URL.

Workflow (Dev → Test → Prod)

  1. Dev: Create feature/<area>-<desc>, implement with Copilot assistance, write unit tests, ensure lint/typecheck pass.
  2. PR to Dev: Open PR to Dev main. Use Copilot Chat to draft PR description and checklists. Review → merge.
  3. Promote to Test: Pull latest Dev; push a branch to the test remote; open PR in RevNova-Test. Run integration tests.
  4. Release to Prod: After QA sign-off, open PR to RevNova (Prod). Tag release on merge.

Copilot tips: Use inline prompts to generate tests, refactor code, and explain changes. Use Copilot Chat to summarize diffs and propose PR content.

For questions, check requirements pages or ask in the team channel. Avoid sharing credentials in chat; use the secrets vault.
